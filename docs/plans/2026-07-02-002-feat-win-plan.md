### Phase D2 — Deploy to Railway

**Goal:** `api.py` (FastAPI backend) + `dashboard/out/` (static export) deployed to Railway's free tier so the demo runs at a public URL.

**Files to add:**
- `projectbrain/Dockerfile` — Multi-stage: build dashboard, run api.py with uvicorn, serve static from port 8000.
- `projectbrain/requirements.txt` — Pin all deps (fastapi, uvicorn, cognee, etc.)
- `projectbrain/railway.json` — Minimal: start command, healthcheck path.

**Contracts:**
- Dockerfile starts uvicorn on `0.0.0.0:8000`
- Static files served via FastAPI's `StaticFiles` at `/` (already in api.py)
- Railway healthcheck hits `/api/metrics` (always returns 200)
- `COGNEE_API_KEY` and `COGNEE_API_URL` set in Railway env vars

**Cognee Cloud compatibility:**
- api.py already uses `cognee_init.py` which reads `COGNEE_API_KEY` + `COGNEE_API_URL` 
- Railway env vars → `os.environ.get()` → works
- No local Cognee config needed — all graph operations go to Cognee Cloud API

**Edge cases:**
- Build fails on Windows paths → use `WORKDIR` + forward slashes in Dockerfile
- `COGNEE_API_KEY` not set → Railway deployment succeeds, but `/api/stream` returns empty graph. Error shown in dashboard SSE banner.
- Static export missing → Railway deploy fails (build step runs `npm run build && npm run export`)

**Ponytail notes:**
- No CI/CD pipeline. Manual `railway up` or push-to-deploy from GitHub.
- No domain setup needed — Railway provides `<project>.railway.app` URL.
- `ponytail: single Dockerfile, no docker-compose. One service handles everything.`

**Verification:**
- `docker build .` succeeds in projectbrain/
- `railway up` deploys without error
- `curl https://<project>.railway.app/api/metrics` returns JSON
- Dashboard loads at the Railway URL
- SSE stream works over Railway's public URL

**Ponytail notes:**

### Phase D5 — Live Graph Growth Demo

**Goal:** A script that incrementally adds knowledge and shows the graph growing via SSE. End-to-end demo choreography.

**Files to create:**
- `projectbrain/demo_seed.py` — Script that adds 1 entry → waits 3s → adds next → waits → loops. Showcases live SSE updates.

**How it works:**
1. `demo_seed.py` posts entries to `POST /api/webhook/remember` one at a time with 3-second delays
2. Each POST triggers `cognee.add()` + `cognee.cognify()` → graph updates → SSE pushes new graph to dashboard
3. Dashboard graphs grows node by node, edge by edge
4. Script prints: `[1/10] Added: Decision: Use Cognee Cloud` so the presenter can narrate

**Contracts:**
- `demo_seed.py` uses local API at `http://127.0.0.1:8000`
- Each entry is a unique Decision/Incident/Metric
- 3-second delay between entries (not too fast, not too slow for demo pacing)
- Script has `--count` flag to limit number of entries

**Edge cases:**
- `POST /api/webhook/remember` fails → script retries once, then prints error and continues
- API server not running → clear error message on first POST
- Ctrl+C → graceful exit (prints "Demo stopped")

**Ponytail notes:**
- `demo_seed.py` is a convenience wrapper around the existing remember endpoint.
- No new API surface. Pure choreography.
- `ponytail: 10 entries total — each a real, unique knowledge item.`

**Verification:**
- Start api.py → start dashboard → run `python demo_seed.py`
- Dashboard graph grows incrementally with each entry
- SSE events fire after each entry
- Metrics update (node count increases, last_cognify updates)
- Demo completes without errors

### Phase D6 — Forget Safety Net

**Goal:** 5-line soft-confirm before forget operations. API now supports a dry-run preview of what a forget would remove.

**Files to change:**
- `projectbrain/api.py` — New endpoint + forget confirmation logic

**New endpoint:**
- `POST /api/forget/preview` — Dry run: accepts `{query: "..."}`, returns what would be removed (node names, IDs)
- `POST /api/forget/confirm` — Accepts `{query: "..."}`, runs `cognee.forget()` only if preview was called first

**Contracts:**
- Forget is a two-step flow: preview → confirm
- Preview returns matching node names/IDs without deleting
- Confirm only works if preview was called within the last 60 seconds (session guard)
- Without preview first, confirm returns `{"error": "Call preview first"}`

**Edge cases:**
- No matches for query → preview returns `{matches: []}`, confirm returns `{deleted: 0}`
- 60s timeout expired → confirm returns `{error: "Preview expired. Call preview again."}`
- `cognee.forget()` fails → endpoint returns error, no state change

**Ponytail notes:**
- Session guard uses a simple `last_preview_time` + `last_preview_query` in memory
- Resets on server restart. Fine for demo.
- `ponytail: in-memory guard only. No DB. Enough for the hackathon.`

**Verification:**
- `POST /api/forget/preview {"query": "test"}` → returns matching nodes
- `POST /api/forget/confirm {"query": "test"}` → deletes and returns `{deleted: N}`
- `POST /api/forget/confirm {"query": "test"}` without preview → error
- Wait 60s+ → confirm returns expired error
- Graph updates after forget → SSE notifies clients → dashboard reflects change

## Risk Register

| Demo video too long (target: 3-4 min) | Medium | Medium | Write script in MASTER.md, time each section |
| Docker build on Windows (newlines, paths) | Medium | Low | Linux-style paths (`WORKDIR`, `COPY` with `./`), CRLF-safe |
| Dashboard build (npm run build) fails in Docker | Medium | High | Use Node 20 LTS in Dockerfile, install dependencies first |
| SSE over Railway WSS/proxy | Low | Medium | Railway supports SSE through HTTP/1.1; test after deploy |
